= telocity Documentation

* *Main commands* – `translate`, `transform`, `oneshot` – for interacting with LLMs.
* *Management commands* – `remove`, `config` – to clean up and configure.
* *Misc commands* – `strip`, `tokencount`, `average`, `merge`, `split`, `batchgen`, `read` – to inspect and prepare text.

All commands share a common philosophy: it's KISS, basic, will always only ever work with plain text (use link:https://github.com/kovidgoyal/calibre[calibre] to convert pagans to the plain text religion), is opinionated and favors a preset way of things and offers little in the way of flexibility (but more in composability). What little can be configured is meant to further the laziness of running the commands with as few arguments as possible. I don't have any hopes for any other person to ever adopt this thing, it's mainly for myself and if someone else finds a use for it, it's a happy accident that someone else's mind would favor the exact same way of doing things as I do!

I am a simpleton, I like simple things, but definitions of simple can vary—by a lot.

Here's an example of opinionated: the terminal output is formatted to 80col and is not, and will never be, user configurable. (editing the source code is free tho')

This documentation was mainly LLM generated by feeding the source code to the (new as of February 2026) DeepSeek, with some adjustments on errors, repetitions and excessive superlative language. If you work with LLMs often you can tell the parts I hand wrote / adjusted and the parts that are pure LLM output. Still, it's LLM writing, which means it's not very nice to look at and at the end of the day I had this doc made as another experiment on using LLMs more than out of belief that it would be actually read by others. Once again, if you appreciate it: well, it's a happy accident.

My o-p-i-n-i-o-n © ™ on this matter: it's workable, but LLMs still have a little too much of that slop smell. I could see myself doing this on another's project that lacks documentation to have some if I needed it, but a project officially using the output directly would not be looked at kindly.

== Quick Overview

[cols="1,1,3", options="header"]
|===
| Category | Command (alias) | Description

| *Main*
| `translate` (`tr`)
| Translate a text file with full control over chunking, batching, and resume.

|
| `transform` (`tf`)
| Generic batch processor for any LLM task (summarization, rewriting, …).

|
| `oneshot` (`os`)
| Single‑shot streaming completions – perfect for interactive use or piping.

| *Management*
| `remove` (`rm`)
| Delete progress files created by long‑running jobs.

|
| `config` (`cfg`)
| View, edit, or reset configuration and language settings.

| *Misc*
| `strip` (`st`)
| Clean text: remove/extract delimiters, compress blank lines, strip Markdown.

|
| `tokencount` (`tc`)
| Count tokens using HuggingFace tokenizers (downloadable).

|
| `average` (`avg`)
| Show average line length (graphemes and bytes).

|
| `merge` (`mg`)
| Recursively merge files with a given extension into one.

|
| `split` (`sp`)
| Split a large file into line‑aware parts of a target size (MB).

|
| `batchgen` (`bg`)
| Generate a JSONL batch file for providers like Mistral, Gemini, OpenAI.

|
| `read` (`rd`)
| Launch a local web server to read a text file as a paginated e‑book.
|===

== Common Concepts

* *State directory* – where configuration, progress files, and downloaded tokenizers live:
** Linux: `~/.config/telocity/`
** Windows: `%APPDATA%\telocity\`
** macOS: `~/Library/Application Support/telocity/`

* *Progress files* – each long‑running job (`translate`, `transform`) creates a `<hash>.json` and a lock file. Use `telocity rm` to delete them when you're done.

* *Presets* – model configurations stored in `config.json`. They define chunk sizes, batch sizes, parallelism, prompts, and more. Select a preset with `--params`.

* *Retries & resume* – failed chunks are retried (with slightly higher temperature). Interrupted jobs can be resumed later from where they left off.

* *First run* – telocity automatically creates a default `config.json` and downloads nothing else. Tokenizers are fetched on demand with `telocity tc --downloadmodel <name>`.
